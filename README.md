# Media Service

Внутренний инфраструктурный сервис для распределенного хранения медиафайлов. Выступает в роли **Control Plane** для управления жизненным циклом блобов, перенося основную нагрузку по передаче данных (загрузка/скачивание) на S3-совместимый **Data Plane**.

## Архитектура

Система построена на основе модели **Content-Addressable Storage (CAS)**.

- **Control Plane (Media Service):** Управляет сессиями загрузки, генерирует S3 Presigned URL, хранит метаданные в PostgreSQL и асинхронно вычисляет SHA256 хэши для дедупликации.
- **Data Plane (S3 / MinIO):** Напрямую принимает файлы от клиентов и отдает их для скачивания по короткоживущим криптографическим ссылкам.
- **Разделение ответственности:** Данный сервис _не_ занимается авторизацией пользователей или проверкой квот. Бизнес-сервисы (например, `posts-service`) сами проверяют права пользователя, после чего безопасно запрашивают URL для загрузки или скачивания у `media-service`.

## Модель безопасности и нюансы Production

Архитектурно хранилище S3 является полностью приватным. Доступ к блобам предоставляется исключительно через **Presigned URLs**, содержащие короткоживущие криптографические подписи (`X-Amz-Signature`). Клиенты не могут угадать пути к файлам благодаря использованию хэширования SHA256.

> **ВАЖНЫЙ НЮАНС ДЛЯ PRODUCTION:**
> В локальном файле `docker-compose.yml` присутствует скрипт инициализации (`mc anonymous set download myminio/media-blobs`), который временно делает бакет публичным. Это сделано **ИСКЛЮЧИТЕЛЬНО для удобства локальной отладки**.
>
> **В Production среде:**
> S3 бакеты ДОЛЖНЫ быть настроены как **Private** (анонимный доступ запрещен). Все запросы на скачивание ДОЛЖНЫ проходить через эндпоинт `GET /blobs/{blobId}/download-url` для получения безопасной, временной inline-ссылки.

## Основные процессы

### Процесс загрузки (Resumable & Multipart)

1. **Инициализация:** Клиент запрашивает загрузку через бизнес-сервис. Media Service вызывает метод `CreateMultipartUpload` в S3 и возвращает S3 Presigned URLs для отправки частей файла (чанков).
2. **Прямая загрузка:** Клиент загружает части файла напрямую в S3, получая в ответ заголовки `ETag`.
3. **Завершение:** Клиент отправляет список `ETags` в Media Service. Media Service завершает процесс Multipart Upload в S3.
4. **Асинхронная дедупликация (HashWorker):** Фоновый воркер скачивает временный объект, вычисляет `SHA256`, проверяет наличие дубликатов в базе и перемещает объект по каноничному CAS-пути (например, `blobs/ab/cd/abcdef...`).

### Процесс просмотра (Inline Viewing)

1. Бизнес-сервис авторизует запрос пользователя на просмотр файла.
2. Бизнес-сервис вызывает `GET /blobs/{blobId}/download-url`.
3. Media Service генерирует Presigned URL с параметром `response-content-disposition=inline` и сроком жизни 1 час.
4. Браузер клиента скачивает медиафайл напрямую из S3, отображая его во встроенном плеере или интерфейсе.

### Сборка мусора (GCWorker)

- Периодически сканирует блобы со статусом `PENDING_REFERENCE`, возраст которых превышает `grace_period` (например, 24 часа).
- Удаляет зависшие объекты из S3 и записи из базы данных, предотвращая утечки дискового пространства от брошенных загрузок.

## Разработка (Makefile)

Требуется Go 1.25+, Docker и [golang-migrate](https://github.com/golang-migrate/migrate).

```bash
# Запустить MinIO, PostgreSQL, применить миграции БД, сгенерировать OpenAPI и запустить сервер
make dev

# Перегенерировать документацию OpenAPI
make swagger

# Остановить и удалить локальное окружение
make docker-down
```

## Документация API

После запуска сервиса, интерактивная документация OpenAPI (Swagger) доступна по адресу:
`http://localhost:8080/swagger/index.html`
